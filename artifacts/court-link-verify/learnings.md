## yeyak rsv_svc_id는 월/보름 단위 회차로 로테이션된다

**상황**: Task 1, 잠원·광나루 평일 ID를 검색 ajax로 실조회하던 중. 모든 회차의 USE 기간이 2026.07.01~07.31(홍은은 7/1~15, 7/16~31 보름 분할)로 잘려 있음을 발견.
**판단**: 등록된 ID는 그 달이 지나면 확실히 죽는다 — "외부 사이트가 언제 바뀔지 모른다"가 아니라 **매달 확정적으로 바뀐다**. 매일 점검 루틴의 가치가 스펙 작성 시점 추정보다 훨씬 높다. 홍은은 보름 단위라 월 2회 교체 필요. courts.ts 주석에 로테이션 사실을 명시했고, Task 5(검색 후보 탐지)가 이 교체를 자동화한다.
**다시 마주칠 가능성**: 높음 — 매달 재발이 보장된 구조. 루틴이 안정화될 때까지 월초 점검 결과를 지켜봐야 함.

## yeyak '저녁' 회차는 등록하지 않음

**상황**: Task 1 실조회에서 평일/주말 외에 '저녁' 회차(별도 ID)가 코트마다 존재함을 발견.
**판단**: spec의 요일유형(평일/주말·공휴일) 모델은 시간대 회차(주간/저녁)와 직교하는 별개 축 — 이번 범위에 넣지 않고 주간 회차만 등록. 필요해지면 BookingLink에 label 축을 추가하는 별도 feature로.
**다시 마주칠 가능성**: 중간 — 사용자가 "저녁 예약이 안 보인다"고 할 수 있음. 그때 이 기록이 출발점.

## yeyak 캘린더 ajax는 WAF가 비브라우저 접근을 차단한다

**상황**: 체크포인트 1에서 yeyak 카드가 "현황 확인 불가"로 뜨는 원인 추적. 앱에 등록된 `selectDayList.do`는 404(가정된 URL이 실제로 존재하지 않음 — tennis-radar plan의 "미검증" 항목이 실제 버그로 확인). 페이지 JS에서 진짜 엔드포인트 `selectListReservCalAjax.do`를 찾았으나, 세션 쿠키+전체 폼+브라우저 헤더를 갖춰도 302 → `/management/ipRedirect.do?threatGb=DNP`("비정상 접근 차단")로 WAF가 차단.
**판단**: (1) 딥링크(selectReservView.do)는 curl로 정상 — 사용자 예약 경로는 문제없음. (2) 잔여 현황 fetch는 서버사이드에서 구조적으로 불가 — 앱은 이미 loadError로 우아하게 처리하므로 이번 feature 범위에서 고치지 않음(스펙: 앱 동작 변경 없음). (3) 검증 targets에서 yeyak 캘린더는 자동 점검 대상이 아니라 "알려진 제약"으로 리포트에 명시 — 매일 거짓 알람을 만들지 않기 위해 (gytennis TLS 오판 방지 불변 규칙과 같은 원리).
**다시 마주칠 가능성**: 높음 — 공공기관 사이트 WAF는 흔함. "데이터 조회가 안 되면 브라우저를 붙이기 전에, 그 데이터가 정말 필요한지·우아한 실패로 충분한지 먼저 판단"이 교훈.

## 외부 페이지 실캡처 픽스처에는 PII가 섞여 들어온다

**상황**: Step 4 code-reviewer가 yeyak 실캡처 픽스처(yeyak-valid.html)에서 실제 개인 휴대폰 번호를 발견 (Critical). 공개 repo에 push되기 전이었음.
**판단**: 스크립트/스타일 제거 + 전화번호·이메일 마스킹으로 정리 (157KB→67KB). "원본이 공개 사이트에 있다"는 사실이 별도 저장소에 아카이브해도 되는 근거가 아니다. 외부 페이지를 픽스처로 캡처할 때는 커밋 전 PII 스캔(전화번호·이메일 정규식)이 필요.
**다시 마주칠 가능성**: 높음 — 외부 사이트 스크레이핑 feature마다 실캡처 픽스처를 만들게 됨.

## gytennis daily 페이지는 심야에 안내 shell을 반환한다

**상황**: 리뷰 반영으로 gytennis에 expectedText를 추가하려고 심야(01시)에 실페이지를 확인하니, 예약 UI가 아니라 "예약 가능한 시간이 아닙니다"(05~22시) 200 shell이 반환됨.
**판단**: expectedText를 코트명으로 잡았으면 심야 루틴마다 거짓 알람이 났을 것. shell에도 포함되는 '예약'을 최소 판별 문자열로 선택. 루틴 실행 시각은 예약 가능 시간대(주간)로 잡는 게 안전.
**다시 마주칠 가능성**: 중간 — 외부 사이트의 시간대별 응답 차이는 검증 자동화에서 반복될 함정.

## 루틴은 클라우드 대신 로컬 스케줄러로 등록 (plan 미결정 항목의 예상 경로)

**상황**: Task 8. 사용자는 클라우드 루틴을 선택했으나, 이 세션에서 쓸 수 있는 스케줄러는 데스크톱 앱 로컬 스케줄러뿐 (클라우드 루틴은 claude.ai/code 웹 UI에서 등록해야 함). plan 미결정 항목에 "불가하면 로컬 스케줄로 전환 (전환 비용 낮음)"으로 이미 예상해둔 갈래.
**판단**: 자동화를 즉시 작동시키는 것이 우선 — 로컬 루틴(매일 09:06, 앱이 닫혀 있었으면 다음 실행 시 보충)으로 등록하고, 클라우드 이전은 사용자 안내로 넘김. 루틴 프롬프트는 저장소 경로·절차·불변 규칙을 전부 담아 자체 완결로 작성 — 클라우드로 옮길 때 프롬프트를 그대로 복사하면 된다.
**다시 마주칠 가능성**: 중간 — 스케줄 자동화가 필요한 feature마다 "어느 스케줄러에 등록 가능한가"는 실행 환경에 따라 달라진다. 아키텍처를 CLI 명령 하나(verify:courts)에 격리해둔 것이 전환 비용을 낮췄다.
