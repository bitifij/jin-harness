# court-link-verify — 코트 링크 검증·수리 체계

## 개요

tennis-radar가 의존하는 외부 예약 사이트 3곳(gytennis, 양평누리, yeyak)의 예약 딥링크와 데이터 조회 경로가 실제로 유효한지 매일 자동 점검하고, 깨진 것을 발견하면 조사해서 고치거나(가능한 경우) 사용자에게서 숨긴다(불가능한 경우). 지난 배포에서 "데이터 조회는 되는데 사용자가 클릭하는 예약 링크는 전부 깨져 있던" 사각지대의 재발을 막는다.

## 범위

### 포함

- 등록된 모든 코트의 **예약 딥링크**와 **데이터 조회 경로**를 한 번에 점검하는 검증 명령 — 운영자가 로컬에서 수동 실행할 수도 있다
- yeyak 소프트 404 감지 — HTTP 200이 떠도 페이지 내용으로 진위를 판정
- yeyak **평일/주말·공휴일 요일유형별 예약 ID**를 데이터 모델에 반영하고, 앱 화면에서 요일유형에 맞는 예약 링크 제공
- 매일 1회 자동 실행되는 점검 루틴 — 깨진 링크를 발견하면 조사해서 **자동 수정 + 커밋**, 수정 불가 시 **앱에서 숨김 + 보고**
- 매 실행마다 갱신되는 검증 리포트 (실행 시각, 코트별 판정, 수정·숨김 내역)

### 제외

- **앱 런타임에서의 실시간 링크 감지** — 매일 주기 점검으로 방치 기간이 최대 하루라 충분하고, 런타임 감지는 사용자 응답 속도를 해친다
- **gytennis TLS 인증서 체인 문제 해결** — 별도 배포 차단 이슈로 이미 기록되어 있음 (learnings.md). 단, 검증 명령은 이 문제와 무관하게 gytennis를 점검할 수 있어야 한다 (불변 규칙 참조)
- **신규 코트 자동 발굴** — 이번 범위는 "등록된 코트의 유효성 유지"로 한정. 발굴은 성격이 다른 feature
- **깨진 링크의 화면 내 경고 배지 등 추가 UX** — 깨진 코트는 숨기는 것으로 충분하다고 결정

## 시나리오

### 1. 전체 정상 — 검증 실행

- **Given** — 등록된 모든 코트의 딥링크·데이터 조회 경로가 유효한 상태
- **When** — 검증 명령을 실행
- **Then** — 코트별 정상 판정과 요약(예: "14개 점검, 14개 정상")이 출력되고, 리포트가 실행 시각과 함께 갱신되며, 명령이 성공 상태(종료 코드 0)로 끝난다

성공 기준:
- [ ] 모든 링크 유효 → 출력에 실패 항목 0건, 종료 코드 0
- [ ] 리포트에 이번 실행 시각과 코트별 판정이 기록된다
- [ ] 요일유형별 ID가 여러 개인 코트는 ID **각각**이 개별 점검 항목으로 나타난다

### 2. 하드 깨짐 감지 — 404·리다이렉트

- **Given** — 어떤 코트의 딥링크가 404를 반환하거나 로그인 페이지로 리다이렉트되는 상태 (지난번 gytennis·양평누리 사례)
- **When** — 검증 명령을 실행
- **Then** — 해당 코트가 실패로 판정되고 원인(상태 코드/리다이렉트 대상)이 출력되며, 명령이 실패 상태(종료 코드 비0)로 끝난다

성공 기준:
- [ ] 404 응답 링크 → 해당 코트 실패 판정 + 원인 표시, 종료 코드 비0
- [ ] 다른 정상 코트들은 여전히 정상으로 판정된다 (한 개 실패가 전체 점검을 중단시키지 않음)

### 3. 소프트 404 감지 — yeyak 가짜 ID

- **Given** — yeyak 예약 ID가 유효하지 않아, 링크 접속 시 HTTP 200이지만 해당 코트의 서비스 정보가 없는 페이지가 뜨는 상태 (지난번 망원·뚝섬·반포 사례)
- **When** — 검증 명령을 실행
- **Then** — HTTP 200임에도 해당 코트가 실패로 판정된다

성공 기준:
- [ ] 200 응답이지만 페이지에 해당 코트를 식별하는 내용이 없음 → 실패 판정
- [ ] 200 응답이고 해당 코트 내용이 있음 → 정상 판정

### 4. ID 변경 감지 — 대체 후보 제시

- **Given** — yeyak에서 같은 시설이 새 예약 ID로 재등록되어, 등록된 ID는 죽고 사이트 검색에는 새 ID가 조회되는 상태
- **When** — 검증 명령을 실행
- **Then** — 해당 코트가 "변경 감지"로 판정되고, 사이트 검색에서 찾은 대체 ID 후보(시설명·회차 정보 포함)가 함께 출력된다

성공 기준:
- [ ] 죽은 ID + 검색에서 동일 시설명 발견 → 변경 감지 판정 + 새 ID 후보가 리포트에 나타난다

### 5. 평일/주말 예약 링크 — 앱 화면

- **Given** — 요일유형별 예약 ID가 각각 등록된 yeyak 코트(예: 잠원한강공원)의 카드
- **When** — 사용자가 예약 링크를 본다
- **Then** — 요일유형별 예약 진입점이 구분되어 표시되고(예: "평일 예약", "주말·공휴일 예약"), 각각 해당 요일유형의 예약 페이지로 연결된다

성공 기준:
- [ ] 요일유형 ID 2개 등록 코트 → 카드에 요일유형별 예약 링크가 각각 표시된다
- [ ] 각 링크는 해당 요일유형의 예약 ID 페이지로 연결된다
- [ ] ID가 1개뿐인 코트(gytennis 등)는 기존처럼 단일 예약 링크가 표시된다

### 6. 루틴 — 자동 수정 + 커밋

- **Given** — 매일 점검 루틴이 실행되고, 깨진 링크가 발견되었으며, 조사(사이트 검색·페이지 확인)로 올바른 대체값이 확실하게 확인된 상태
- **When** — 루틴이 완료된다
- **Then** — 코트 데이터가 새 값으로 수정되어 커밋되고, 리포트에 "무엇이 왜 어떻게 바뀌었는지"가 기록되며, 수정된 링크는 재검증을 통과한 상태다

성공 기준:
- [ ] 수정 커밋의 변경 내역과 리포트 기록이 일치한다
- [ ] 수정된 값으로 검증 명령을 다시 실행하면 해당 코트가 정상 판정된다

### 7. 루틴 — 수정 불가 시 숨김 + 보고

- **Given** — 깨진 코트를 조사했지만 대체값을 찾지 못한 상태 (시설이 사이트에서 사라진 경우 등)
- **When** — 루틴이 완료된다
- **Then** — 해당 코트가 앱 화면에서 보이지 않게 되고(코트 데이터 자체는 보존), 리포트에 숨김 사유가 기록되며, 변경이 커밋된다

성공 기준:
- [ ] 숨김 처리된 코트 → 앱 코트 목록에 나타나지 않는다
- [ ] 숨김 처리된 코트의 데이터는 삭제되지 않고 남아 있다 (복구 가능)
- [ ] 리포트에 숨김 사유와 시각이 기록된다

## 불변 규칙

- **데이터 일관성**: 자동 수정은 **재검증을 통과한 값만** 커밋된다 — 루틴이 스스로 고친 값이 또 깨진 값이면 안 된다.
- **데이터 일관성**: 사용자가 앱에서 클릭할 수 있는 모든 외부 링크는 빠짐없이 검증 대상에 포함된다 — 검증 목록과 앱 노출 링크가 어긋나면 이번 사각지대가 재발한다.
- **판정 신뢰성**: 검증 판정은 실제 대상 사이트의 실시간 응답에만 근거한다. gytennis의 TLS 문제(Node/Bun fetch 실패)가 검증 실패로 오판되지 않아야 한다 — 사이트가 브라우저/curl로 정상이면 정상으로 판정한다.
- **안전성**: 기존 테스트·빌드(`bun run test`, `bun run build`)는 이 feature 이후에도 외부 네트워크 없이 통과한다 — 외부 접속은 검증 명령·루틴에만 격리된다.
- **보존성**: 루틴은 코트 데이터를 삭제하지 않는다. 제거가 필요하면 숨김까지만 하고 최종 삭제는 사람이 결정한다.

## 의존성

- tennis-radar feature (머지 완료) — 코트 데이터, 카드 UI, 소스 파서
- 외부 사이트 3곳: gytennis.or.kr, srent.y-sisul.or.kr(양평누리), yeyak.seoul.go.kr
- yeyak 검색 경로 — 시설명으로 예약 ID를 조회하는 방법은 이미 확인됨 (learnings.md 2026-07-09: 검색 ajax를 브라우저 없이 직접 호출 가능)
- 매일 자동 실행을 위한 스케줄 실행 환경 (루틴)

## 미결정 항목

- 없음 — 제기된 질문(다중 ID 반영 여부, 자동화 수준, 수정 불가 시 처리, 결과 확인 방식, 주기)은 모두 결정됨.
- 참고(저비용 기본값): 요일유형별 예약 링크의 표시 방식은 "카드에 링크 2개 병렬 표시"를 기본값으로 제안 — 시각 디테일은 구현 시 조정 가능.
